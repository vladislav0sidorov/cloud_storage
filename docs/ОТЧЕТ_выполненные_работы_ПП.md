# Выполненные работы (преддипломная практика / DevOps) — тезисы для отчёта

Ниже по пунктам то, что реализовано в репозитории `cloud_storage`, чтобы перенести формулировки в отчёт.

---

## 1. Контейнеризация серверной части (backend)

- Добавлен **`cloud-server/Dockerfile`**: образ на базе **Node.js 22** (вариант **bookworm-slim** для совместимости с нативным модулем **bcrypt** без сборки под Alpine).
- Установка зависимостей через **`npm ci --omit=dev`**, запуск приложения командой **`npm run start`** (**`node index.js`**).
- В образе создаются каталоги **`uploads/tmp`** для работы **multer** при загрузке файлов.
- Добавлен **`cloud-server/.dockerignore`**: исключены `node_modules`, `uploads`, `.env`, служебные файлы — уменьшение контекста сборки и исключение утечки секретов в образ.

---

## 2. Контейнеризация клиентской части (frontend)

- Добавлен **`cloud-client/Dockerfile`** в виде **multi-stage**:
  - стадия **build**: установка зависимостей, **`npm run build`** (Vite + TypeScript);
  - стадия **runtime**: образ **nginx:alpine**, раздача статики из **`dist`**.
- Время сборки задаются **`ARG` / `ENV`**: **`VITE_API_URL`**, **`VITE_WS_URL`** (подстановка публичных URL API и WebSocket для продакшена).
- Добавлен **`cloud-client/nginx.conf`**: раздача SPA (**`try_files`** → **`index.html`**), кэширование для **`/assets/`** при необходимости.
- Добавлен **`cloud-client/.dockerignore`**.

---

## 3. Локальный запуск стека (docker-compose в корне репозитория)

- Файл **`docker-compose.yml`** в корне: сервисы **`mongo`** (образ **mongo:7**, том **`mongo_data`**, **healthcheck**), **`api`** (сборка из **`./cloud-server`**), **`web`** (сборка из **`./cloud-client`**).
- Том **`uploads_data`** для каталога загрузок на API.
- Переменная **`MONGODB_URI`** для сервиса API в compose переопределяет значение из **`.env`**, чтобы в Docker использовалась **MongoDB внутри сети compose** (`mongodb://mongo:27017/...`), а не облачный кластер из файла разработчика.
- Переменная **`CLIENT_URL`** для CORS под сценарий «фронт на порту 80», а не на **3000** (через **`DOCKER_CLIENT_URL`** или значение по умолчанию **`http://localhost`**).

---

## 4. Доработки backend для production-образа

- В **`cloud-server/package.json`**: скрипт **`"start": "node index.js"`** для запуска без **nodemon**.
- Зависимость **`dotenv`** перенесена из **devDependencies** в **dependencies**, чтобы при **`npm ci --omit=dev`** не ломался импорт **`import 'dotenv/config'`**.

---

## 5. Исправление сборки клиента (для прохождения CI / Docker build)

- Устранена зависимость от отсутствующего модуля **`@/features/Chat`** в хуке **`useChatSocket`** (файл удалён или упрощён — в актуальном состоянии репозитория хук не блокирует **`tsc` / `vite build`**).

---

## 6. Публикация приложения на VPS (каталог `deploy/`)

- **`deploy/docker-compose.yml`**: продакшен-сценарий **без сборки на сервере** — подтягиваются готовые образы из **GHCR** по префиксу **`${GHCR_PREFIX}`** (`cloud-server`, `cloud-client`), **MongoDB** в том же compose, тома **`mongo_data`**, **`uploads_data`**.
- **`deploy/env.example`**: шаблон переменных для файла **`.env`** на сервере (**`GHCR_PREFIX`**, **`CLIENT_URL`**, **`API_URL`**, секреты JWT, почта, **`MONGODB_URI`** при использовании Atlas и т.д.).
- **`deploy/README.md`**: пошаговая инструкция — установка **Docker** на **Ubuntu**, **UFW** (в т.ч. нестандартный SSH-порт), вход в **GHCR**, копирование **`deploy`**, первый запуск **`docker compose pull` / `up`**, замечания по домену и HTTPS, бэкапы томов, раздел **автодеплоя** (см. п. 8).

---

## 7. Непрерывная интеграция (CI) — GitHub Actions

- Файл **`.github/workflows/ci.yml`**.
- **Job «Client»**: **`npm ci`**, **`npm run lint`**, **`npm run build`** с подстановкой **`VITE_*`** из **переменных репозитория** (**`vars.VITE_API_URL`**, **`vars.VITE_WS_URL`**) с запасными значениями для локального URL.
- **Job «Server»**: **`npm ci`**, проверка синтаксиса **`node --check index.js`**.
- **Job «Docker»**: перед **`docker compose build`** создаётся **временный** **`cloud-server/.env`** (реальный файл в **`.gitignore`**), чтобы compose не падал из‑за отсутствия **`.env`** в CI.
- **Concurrency**: отмена устаревших прогонов при новых push в ту же ветку.

---

## 8. Непрерывная доставка образов (CD) в GHCR

- **Job «push-images»** (только ветка **`main`**, события **`push`** и **`workflow_dispatch`**): логин в **`ghcr.io`** через **`GITHUB_TOKEN`**, сборка и публикация образов **`cloud-server:latest`** и **`cloud-client:latest`** с префиксом **`ghcr.io/<owner>/<repo>/...`** (имя репозитория в **нижнем регистре**).
- Префикс образов передаётся в теги через **шаг с `id: ghcr`** и **`GITHUB_OUTPUT`** (**`steps.ghcr.outputs.prefix`**), чтобы не использовать **`env.IMAGE_PREFIX`** из **`$GITHUB_ENV`** (устранение предупреждения линтера о недопустимом контексте **`env`**).
- Кэш слоёв Docker: **`cache-from` / `cache-to: type=gha`**.

---

## 9. Автоматическое обновление на VPS после публикации образов

- **Job «deploy-vps»** выполняется при **`vars.VPS_DEPLOY == 'true'`**, после успешного **`push-images`**.
- Подключение к серверу по **SSH** (**`appleboy/ssh-action`**): на VPS выполняются **`docker compose pull`**, **`docker compose up -d --remove-orphans`**, **`docker image prune -f`** в каталоге **`VPS_DEPLOY_PATH`** (по умолчанию **`/opt/cloud_storage/deploy`**).
- Опционально: секрет **`GHCR_READ_TOKEN`** (PAT с **`read:packages`**) для **`docker login`** на сервере, если образы в GHCR **приватные**; для **публичных** образов шаг логина пропускается.
- В отчёте указать **переменные** (**`VPS_HOST`**, **`VPS_PORT`**, **`VPS_USER`**, **`VPS_DEPLOY_PATH`**) и **секрет** **`VPS_SSH_PRIVATE_KEY`** (отдельный ключ для CI, не пароль root).

---

## 10. Что описать в тексте отчёта отдельно (словами)

- **Цель**: автоматизация сборки, хранения артефактов в реестре и выкладки на VPS без ручной сборки на сервере.
- **Схема**: разработчик → **Git** → **GitHub Actions** → **GHCR** → **VPS** (**Docker Compose**).
- **Безопасность**: секреты только в **GitHub Secrets** и в **`.env` на сервере**; в репозиторий не коммитить пароли и ключи; для SSH-деплоя — **отдельная пара ключей**, ограничение прав, при необходимости смена пароля после утечки в открытый канал.
- **Ограничения / перспективы**: один сервер; при смене домена/IP — обновление **`VITE_*`** и пересборка образа **web**; дальнейшее развитие — **HTTPS** (Nginx/Caddy + Let’s Encrypt), отдельный reverse proxy, мониторинг, бэкапы по расписанию.
